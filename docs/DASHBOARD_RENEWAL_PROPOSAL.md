# 新ダッシュボード提案書

## 1. 目的
- 見積を予算、注文書を実績として扱い、予実管理を経営判断に使えるレベルまで引き上げる。
- 売上だけでなく、粗利、仕入、資金繰り、工数、稼働率、空き枠を同じ画面体系で把握できるようにする。
- 月次比較、推移、異常値、注力案件を可視化し、経営分析と営業戦略に直結するダッシュボードへ再設計する。

## 2. 現状調査サマリ

### 2.1 現在できていること
- `DashboardController` で `予算=見積`、`実績=受注確定済み見積` の月次集計は実装済み。
- 売上、粗利、仕入、計画工数、キャッシュフローの基礎集計は存在する。
- `Orders/Index` には別系統で資金繰りダッシュボードがあり、ハードウェアと人件費を分けた月次表示もある。
- 承認タスク、取引先同期、売上ランキングはダッシュボード上に載っている。

### 2.2 現状の問題
- ダッシュボード本体と注文書一覧に似た集計ロジックが分散しており、将来の改修で不整合が起きやすい。
- ダッシュボードは表中心で、推移グラフがほぼない。経営判断向けの視認性が不足している。
- `docs/DASHBOARD_UI_SETTING.md` では `総合 / 開発 / 仕入れ販売 / 保守` のセクション分割を想定しているが、現UIには反映されていない。
- AI分析、営業示唆、異常検知、先月比較コメントは未実装。
- 工数は「計画工数」のみで、担当者別の負荷、山積み、空き枠、月内の偏りが見えない。
- 日報APIを扱う下地コードはあるが、現在のダッシュボード出力にはつながっていない。

### 2.3 工数設定のズレ
- 現在の月間キャパは `config/app.php` の `APP_MONTHLY_CAPACITY_PERSON_DAYS` で管理され、デフォルトは `80` 人日。
- 要件では `1人月=20日`、`1人日=8時間`、`スタッフ10人` なので、全社キャパは `200人日/月`、`1600時間/月` が基本。
- つまり現状のキャパ設定は要件の半分以下で、稼働率や空き工数の見え方がズレる。

## 3. 要件との差分

### 3.1 満たしている点
- 見積を予算、注文書を実績として扱う概念は概ね実装済み。
- 売上、粗利、仕入、工数、資金繰りの基礎データは取れる。

### 3.2 足りていない点
- 経営分析に使うための時系列グラフ群が不足。
- 全社視点と事業別視点が分離されていない。
- 工数の空き状況を「誰が、いつ、どのくらい空いているか」まで掘れない。
- 稼働率の高低に対するアラートや色分け基準が弱い。
- 先月対比、予算差異、資金ショート予兆などのコメントを自動説明できていない。
- 営業戦略につながる「案件構成」「粗利率の悪化案件」「失注/未受注の偏り」などが見えない。

## 4. 再設計方針

### 4.1 画面の考え方
- 1枚のトップ画面で全部を詰め込むのではなく、`経営サマリー` を最上段に置き、その下をセクション分割する。
- セクションは以下の4つを基本とする。
  - 総合
  - 開発
  - 仕入れ販売
  - 保守

### 4.2 データ定義
- 予算: 見積
- 実績: 注文書、または `is_order_confirmed = true`
- 売上発生月: 納期基準
- 仕入支出月: 発注または見積起票基準。ただし将来は正式な発注日カラムへ寄せる
- 回収月: 請求予定日または入金予定日基準
- 工数予算: 見積明細
- 工数実績: 日報連携を使う。未連携期間は暫定で計画工数のみ

### 4.3 実装方針
- ダッシュボードと注文書一覧に散っている集計を `ManagementMetricsService` のような共通サービスへ統合する。
- UI側は同じ月次データセットから、カード、グラフ、表、AI分析文を生成する。
- セクション別集計は `business_division` や保守判定ルールに基づいて同じサービス内で切り出す。

## 5. 新ダッシュボード案

### 5.1 上段: 経営サマリー
- KPIカード
  - 売上 予算 / 実績 / 差異
  - 粗利 予算 / 実績 / 差異
  - 粗利率
  - 仕入 予算 / 実績 / 差異
  - ネットキャッシュフロー
  - 稼働率
- サブ指標
  - 今月受注件数
  - 受注残
  - 高粗利案件比率
  - 低粗利案件件数

### 5.2 推移グラフ
- 月次売上推移: 予算線 + 実績棒
- 月次粗利推移: 予算線 + 実績棒
- 月次資金繰り推移: 支払、回収予定、回収実績、ネットCF
- 月次工数推移: 計画工数、実績工数、キャパ、空き工数
- 粗利率推移

### 5.3 セクション別タブ
- `総合`
  - 全社KPI、月次予実、キャッシュ推移、AI分析
- `開発`
  - 売上、粗利、工数、稼働率、空き工数、担当別負荷
- `仕入れ販売`
  - 売上、粗利、仕入、資金先行負担、回収サイト、在庫リスク
- `保守`
  - 保守MRR、継続率、保守粗利、保守工数、顧客別採算

### 5.4 工数ボード
- 全社キャパ: `10人 x 20人日 = 200人日/月`
- 全社時間キャパ: `200人日 x 8時間 = 1600時間/月`
- 表示項目
  - 今月計画工数
  - 今月実績工数
  - 稼働率
  - 空き工数
  - 超過工数
  - 部門別工数配分
  - 担当者別負荷ランキング
  - 月内の空き枠ヒートマップ
- 判定の目安
  - 70%未満: 余力大
  - 70%〜85%: 健全
  - 85%〜100%: 高稼働
  - 100%超: 過負荷

### 5.4.1 個人別の空き状況を出すための前提
- 現状のデータ構造では、個人別の空き状況は正確には出せない。
- 理由:
  - 見積には案件全体の工数はあるが、誰が担当するかの割当情報がない。
  - そのため、現状は全社または部門全体のキャパ比較までしかできない。
- 個人別工数管理のために必要な追加情報:
  - 見積または見積明細に、複数の担当者を割り当てられること
  - 各担当者ごとの配分率または配分工数を持てること
  - 配分された工数を月次・担当者別に集計できること

### 5.4.2 複数人アサイン時の工数ロジック
- 基本ルール:
  - 工数は `人数倍` ではなく `按分` とする
  - 見積に記載された工数総量は変えない
- 例:
  - `ソフト開発 1人月` を3人に割り当てた場合
  - 誤: `1人月 x 3人 = 3人月`
  - 正: `1人月 ÷ 3人 = 0.333人月ずつ`
- 推奨方式:
  - 初期値は均等按分
  - 必要に応じて手動按分できるようにする
  - 例: A 50%、B 30%、C 20%
- この考え方により、見積工数と配員計画の整合性を保ったまま、個人別の空き状況や過負荷を可視化できる。

### 5.4.3 推奨データ設計
- 個人別空き状況を将来出したいなら、担当者アサインは見積ヘッダ単位ではなく明細単位で持つべき。
- 理由:
  - 実案件では設計、実装、テスト、保守で担当者が異なることが多い
  - 案件全体に対する一括担当では、実態に合った負荷配分を表現しづらい
- 推奨構成:
  - 見積明細
  - 見積明細担当者アサイン
  - 担当者ごとの配分率または配分工数
- この構成なら、将来的に以下が可能になる
  - 担当者別の空き状況
  - 担当者別の月次稼働率
  - 案件横断の山積み確認
  - 部門内の再配員シミュレーション

### 5.5 資金繰りボード
- 今月支払予定
- 今月回収予定
- 今月回収実績
- 3ヶ月先までのネットCF予測
- 資金先行案件ランキング
- 回収遅延注意案件

### 5.6 経営分析AI
- 表示場所: ダッシュボード最上段または右カラム
- 入れる内容
  - 先月と今月の比較サマリ
  - 前年同月比と年初来累計の比較
  - 差異の大きい要因トップ3
  - 粗利悪化の要因
  - 稼働率の過不足
  - 資金繰りリスク
  - 営業アクション提案
- 例
  - 今月は売上が予算比+12%だが、粗利率は-4.2ptで悪化。ハード比率上昇と低粗利案件集中が要因。
  - 来月は開発工数が92%まで積み上がる一方、保守チームは63%で余力あり。前倒し調整候補を提示。
  - 回収超過サイト案件が3件あり、翌月ネットCFがマイナス見込み。

## 7. データ基盤の改善提案

### 7.1 優先度高
- ダッシュボード集計と注文書一覧集計を共通サービスへ統合する。
- 事業区分別の切り分けルールを明文化する。
- 月次キャパを固定値ではなく `スタッフ数 x 20` で算出できるようにする。
- `APP_MONTHLY_CAPACITY_PERSON_DAYS` を暫定値ではなく、設定画面または組織マスタ管理へ寄せる。

### 7.2 優先度中
- 工数実績に日報APIを正式接続し、計画対実績を出す。
- 発注日、請求予定日、入金予定日を明示的に管理し、資金繰り精度を上げる。
- 保守売上と保守工数の判定ルールを固定する。

### 7.3 優先度中
- AI分析用に月次集計テーブルまたはキャッシュテーブルを持つ。
- LLMに渡すためのサマリJSONを別途生成する。

### 7.4 ローカル検証用データ
- ダッシュボード改修は、ローカルでゼロ件データだと判断しづらい。
- そのため、2025年1月〜2026年12月の確認用ダミーデータを固定シーダーで投入できる状態を維持する。
- ダミーデータには以下を含める。
  - 開発案件の予算/実績
  - 仕入れ販売案件の予算/実績
  - 保守スナップショット
  - 当月アラート確認用の未達/過負荷ケース
- これにより、ローカルで以下を毎回確認できる。
  - 4タブの切替表示
  - 売上/粗利/仕入/工数カード
  - 推移グラフ
  - アラート
  - ランキング
  - 工数キャパ `10人 x 20人日`

## 8. 画面構成案

### 8.1 1画面目: 経営ダッシュボード
- 役員・管理者向け
- KPI、推移、AI分析、アラート、資金繰り
- 必須比較軸
  - 当月 vs 予算
  - 当月 vs 前年同月
  - 年初来累計 vs 前年同期間

### 8.2 2画面目: 工数ダッシュボード
- 部門長・PM向け
- 月次キャパ、担当者別負荷、空き枠、配員調整

### 8.3 3画面目: 営業ダッシュボード
- 営業責任者向け
- 見積件数、受注率、平均単価、粗利率、顧客別売上、失注理由

## 9. 段階的な実装提案

### Phase 1: 整合性回復
- 集計ロジックを共通サービス化
- キャパ設定を10人前提に補正
- ダッシュボードと注文書一覧の数値定義を統一
- タブUIを追加して `総合 / 開発 / 仕入れ販売 / 保守` を実装

### Phase 2: 可視化強化
- 表中心からグラフ中心へ変更
- 売上、粗利、仕入、CF、工数の時系列グラフを追加
- 担当者別負荷と空き状況を追加

### Phase 3: AI経営分析
- 先月比 / 予算比の分析コメントを自動生成
- 異常値、過負荷、低粗利案件、CF悪化のアラートを出す
- 営業と経営向けの打ち手を提示

## 10. 優先度付きの提案結論
- 最優先は「数値定義の一本化」と「工数キャパ設定の是正」。
- 次に「総合 / 開発 / 仕入れ販売 / 保守」の4タブ化。
- その次に「グラフ化」と「担当者別工数可視化」。
- 最後に「AI分析」と「営業戦略提案」の自動化。

## 11. 推奨アクション
1. `APP_MONTHLY_CAPACITY_PERSON_DAYS` を暫定で `200` に変更する。
2. ダッシュボードと注文書一覧の集計を共通サービスへ統合する。
3. 新UIは `総合` タブから先に作り、次に `開発` タブを実装する。
4. 工数実績を使うなら、日報API連携の再有効化または正式設計を先に決める。
5. AI分析は、数値定義が固まってから追加する。
6. 個人別空き状況までやるなら、見積明細ごとの複数担当者アサインと按分ルールを先に導入する。

# Firebase リビルド 要件定義書

## 1. 目的

現行 RAKUSHIRU Cloud の販売管理、見積、承認、請求、予実、保守売上、商品マスタ、AI補助機能を、Firebase を基盤とする新システムとして再構築する。

新システムでは、現行業務の主要フローを維持しながら、データ構造、外部連携、権限、同期方式を再設計し、今後の運用変更に耐えやすい構成にする。

## 2. 成功条件

- 現行の主要業務フローを Firebase 上で再現できる。
- 見積作成から承認、受注確定、注文書/検収書、請求連携候補、予実集計までのデータが一貫して扱える。
- Money Forward API 連携と顧客 API 連携は、現行踏襲ではなく「新システムで再検討」として分離されている。
- Firebase Auth、Firestore、Cloud Functions、Cloud Storage、Hosting の責務が明確である。
- 既存データ移行時に、見積番号、顧客、担当者、商品、承認履歴、受注確定状態、金額、工数、保守売上スナップショットを欠落させない。

## 3. 対象範囲

### 3.1 対象機能

- 認証、ログイン、ユーザー管理
- ダッシュボード
- 見積作成、編集、複製、削除
- 見積一覧、検索、フィルタ、一括操作
- 承認フロー
- 受注確定、受注取消
- 失注、フォロー期限管理
- 注文書プレビュー
- 検収書プレビュー
- ローカル請求作成、編集
- 請求一覧
- 商品マスタ、商品分類
- 事業区分別集計
- 経営ダッシュボード、予実、工数、資金繰り集計
- 保守売上管理、月次スナップショット、手修正
- 管理画面の工数基準設定
- 更新履歴通知
- ヘルプ、API連携説明
- 要件定義書からの AI 見積ドラフト生成
- 営業AIコーチ
- 外部向け受注確定見積 API 相当の連携口

### 3.2 対象外

- Money Forward API 連携方式の確定
- 顧客 API / 社員 API の連携方式の確定
- 本番移行手順の確定
- Firebase プラン、課金上限、監視設計の最終決定
- UI デザインの全面刷新

## 4. 現行システム概要

現行システムは Laravel 12 + Inertia/React + MySQL で構成される社内向け販売管理システムである。

主な現行データは次のとおり。

| 領域 | 現行データ |
| --- | --- |
| 見積 | `estimates.items` に明細 JSON、`approval_flow` に承認ステップ、`is_order_confirmed` に受注確定状態 |
| 顧客 | `partners` に Money Forward 取引先をローカル保存 |
| 商品 | `products` と `categories` に商品、分類、事業区分、原価、MF品目ID |
| 請求 | `billings` に Money Forward 請求、`local_invoices` に自社請求 |
| 保守売上 | `maintenance_fee_snapshots` と `maintenance_fee_snapshot_items` に月次値 |
| 認証 | Laravel ユーザー、外部ユーザー ID、個別工数キャパ |
| 外部連携 | Money Forward OAuth、外部顧客/社員 API、Google Drive、OpenAI、Google Chat、受注確定見積 API |

## 5. 利用者と権限

| 利用者 | 主な操作 |
| --- | --- |
| 一般ユーザー | 見積作成、編集、申請、受注確認、一覧確認 |
| 承認者 | 承認待ち見積の確認、承認、差戻し相当の操作 |
| 管理者 | 工数基準、ユーザー別キャパ、商品分類、運用設定の管理 |
| 経営/管理担当 | ダッシュボード、予実、工数、事業区分、保守売上の確認 |
| 外部連携先 | 受注確定済み見積の取得 |

権限は Firebase Auth の UID を基準に、Firestore のユーザープロファイルとカスタムクレームを併用して管理する。

## 6. 業務要件

### 6.1 認証

- メールアドレスとパスワードでログインできる。
- 将来的に Google Workspace 認証へ拡張できる。
- ユーザーには表示名、メールアドレス、外部ユーザーID、月間稼働人日、権限を保持する。
- 非業務ユーザーや補助アカウントは集計対象から除外できる。

### 6.2 ダッシュボード

- ログインユーザーの承認待ちタスクを表示する。
- 年月指定で総合、開発、仕入れ販売、保守の予実を表示する。
- 売上、粗利、原価、工数、キャパ、資金繰り、顧客別/担当者別ランキングを表示する。
- AI分析結果を表示できる。
- 現行の Money Forward 取引先自動同期は、新システムでは再検討対象とする。

### 6.3 見積

- 見積を新規作成、編集、複製、削除できる。
- 見積番号は人が読める形式で採番する。
- 見積には顧客、顧客担当者、案件名、発行日、期限日、着手日、納品日、納品場所、備考、社内メモ、検収書備考を保持する。
- 明細には商品、数量、単位、単価、原価、税区分、事業区分、説明、工数、納品日を保持する。
- 見積ステータスは `draft`、`pending`、`sent`、`lost` を最低限扱う。
- 受注確定は承認済み見積のみ可能にする。
- 受注確定時は着手日と納品日を必須または確認対象にする。
- 設計/開発明細を含む場合は要件定義書 URL または添付を必須にする。

### 6.4 承認

- 見積ごとに承認フローを保持する。
- 承認フローは複数ステップ、承認者名、承認者ID、状態、承認日時を保持する。
- 現在の承認者だけが承認操作できる。
- 粗利率などの業務ルールにより、承認者追加や社内メモ必須を制御できる。
- 承認済みになると Money Forward 連携候補、受注確定、帳票出力が利用可能になる。

### 6.5 失注・フォロー管理

- 未受注の見積は失注にできる。
- 失注理由、メモ、失注日を保持する。
- 期限超過見積に対し、フォロー期限や判断メモを記録できる。
- 受注確定済み見積は失注に変更できない。

### 6.6 帳票

- 見積プレビュー、注文書プレビュー、検収書プレビューを表示できる。
- 検収書は受注確定済み見積のみ表示できる。
- 検収書には専用備考を使い、対外見積備考を自動流用しない。
- PDF生成方式は Firebase Storage に保存する方式、またはクライアント印刷方式を比較して決める。

### 6.7 請求

- 見積からローカル請求を作成できる。
- ローカル請求は編集、削除、PDF表示相当の操作ができる。
- Money Forward 請求データを新システムへ同期するかは再検討する。
- 現行の `billings` と `local_invoices` の二重ソースは、重複計上を避けるルールを維持する。

### 6.8 商品マスタ

- 商品分類を作成、編集、削除できる。
- 商品を作成、編集、削除、無効化できる。
- 商品コードは分類コードと連番から採番する。
- 商品には価格、原価、単位、税区分、事業区分、説明を保持する。
- 事業区分は第1種から第6種を扱い、予実・工数計算に使用する。
- Money Forward 品目同期は新システムで再検討する。

### 6.9 事業区分・予実・工数

- 見積と請求をもとに、総合、開発、仕入れ販売、保守の実績/予定を集計する。
- 第1種事業の金額は開発予算から除外できる。
- 工数は人日、人月、人時/時間を人日に換算する。
- 着手日から納品日までの月跨ぎ工数は月次配賦する。
- ユーザー別の月間キャパを設定し、工数負荷に反映する。

### 6.10 保守売上

- 月別の保守売上スナップショットを保持する。
- 顧客名、保守料金、ステータス、サポート種別、データソースを保持する。
- 当月は再同期と手修正を併用できる。
- 過去月は保存済みスナップショットを優先し、過去実績が自動で書き換わらないようにする。
- 現行の外部顧客 API からの取得方式は新システムで再検討する。

### 6.11 AI機能

- Google Drive などにある要件定義書をもとに、要件概要、機能要件、非機能要件、未確定要件、備考案を抽出できる。
- 抽出結果から見積明細ドラフトを生成できる。
- 備考生成では顧客、案件名、明細、要件概要、要件定義書 URL を文脈として使う。
- AI失敗時は上限超過や認証エラーなどを区別して表示する。
- OpenAI API 利用はサーバー側の Cloud Functions 経由に限定する。

### 6.12 更新履歴・ヘルプ

- バージョンごとの更新履歴を表示できる。
- 未読の大型更新はログイン後に通知できる。
- ヘルプでは画面別の操作説明と API 連携仕様を確認できる。

### 6.13 外部向け API

- 受注確定済み見積を外部システムへ提供できる。
- 一覧、詳細、差分取得を提供する。
- 返却対象は受注確定済み、かつ削除扱いでない見積に限定する。
- 開発予算では `development_subtotal_excluding_tax` 相当を利用し、第1種を除外する。
- 認証方式は現行の Bearer 固定トークンをそのまま採用せず、新システムで再検討する。

## 7. 外部連携要件

### 7.1 Money Forward API 連携

現行では以下が実装されている。

- 取引先同期
- 品目同期
- 見積同期
- 見積作成
- 見積 PDF 表示
- 見積から請求への変換
- 請求一覧同期
- 請求 PDF 取得
- ローカル請求から Money Forward 請求作成
- OAuth トークン保存とリフレッシュ

ただし、新システムでは Money Forward API 連携は再検討事項とする。

新システムで検討すべき論点は次のとおり。

- Firebase Cloud Functions から Money Forward API を呼ぶか
- OAuth トークンを Firestore、Secret Manager、別バックエンドのどこで保持するか
- ユーザー単位 OAuth を維持するか、管理アカウント連携に寄せるか
- 自動同期を維持するか、手動同期に限定するか
- 見積/請求の正本を Money Forward と新システムのどちらに置くか
- 同期失敗、削除、重複計上をどう扱うか

### 7.2 顧客 API / 社員 API 連携

現行では `https://api.xerographix.co.jp` 系の API から顧客、保守費用、社員情報を取得している。

ただし、新システムでは顧客 API / 社員 API 連携は再検討事項とする。

新システムで検討すべき論点は次のとおり。

- 顧客マスタを Firebase 側で正本化するか
- 既存 API を継続利用するか
- CRM/社内基幹側を正本にするか
- 保守費用の月次スナップショットだけを取り込むか
- 社員情報と Firebase Auth ユーザーをどう紐付けるか

### 7.3 Google Drive / OpenAI / Google Chat

- Google Drive 連携は要件定義書選択と AI解析のために利用している。
- OpenAI API は要件抽出、見積ドラフト、備考生成、ダッシュボード分析に利用している。
- Google Chat は承認通知用途として利用している。
- これらも Cloud Functions 経由に集約し、フロントエンドに API キーや Webhook URL を出さない。

## 8. 非機能要件

| 分類 | 要件 |
| --- | --- |
| セキュリティ | API キー、OAuth トークン、Webhook URL はクライアントに露出しない |
| 権限 | Firestore Security Rules と Cloud Functions の両方で権限チェックする |
| 監査 | 承認、受注確定、失注、外部同期、手修正は履歴を残す |
| 可用性 | 外部 API 障害時もローカル保存済みデータを閲覧できる |
| 性能 | 一覧はページング、検索用インデックス、集計キャッシュを使う |
| 保守性 | 外部連携は Adapter として分離し、MF/顧客APIの変更を業務ロジックへ波及させない |
| データ保全 | 月次スナップショット、見積番号、承認履歴は移行時に検証する |
| ログ | Cloud Logging に操作ログ、同期ログ、AI失敗ログを出す |

## 9. 移行要件

- MySQL の主要テーブルから Firestore へ移行する。
- 見積 `items`、`approval_flow`、`structured_requirements` は JSON 構造を維持して移行する。
- Money Forward の ID は履歴参照用に保持するが、新システムでの再連携にはそのまま使える前提にしない。
- `mf_tokens` は移行対象外とし、再認証または新方式へ切り替える。
- 受注確定済み見積 API の利用先がある場合は、旧 API と新 API の並行稼働期間を設ける。

## 10. 受け入れ基準

- Firebase 環境でログイン、見積作成、承認、受注確定、一覧確認が一通りできる。
- 現行データから移行した見積の金額、税額、工数、承認状態、受注状態が一致する。
- 保守売上スナップショットの月別合計が現行と一致する。
- 外部連携を停止した状態でも、既存データの閲覧と内部業務は継続できる。
- Money Forward API と顧客 API は「再検討事項」として設計上分離され、業務機能が特定連携方式に依存していない。

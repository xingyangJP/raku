# 保守売上管理ブラッシュアップ提案

> 作成日: 2026-03-22  
> 対象画面: `/maintenance-fees`  
> 目的: 要件と現仕様の差分を整理し、優先度つきで改善案を確定する。

## 結論

保守売上管理は「月次スナップショットを持つ」という軸はできていますが、現状は `表示の信頼性` と `運用の証跡` が弱いです。  
先にやるべきことは次の 3 つです。

1. 画面上で「何の数値を見ているか」を明示する
2. 再同期と手修正の履歴を残す
3. ダッシュボード・見積側と同じロジックを 1 箇所へ寄せる

## 1. 確認した要件

`docs/MAINTENANCE_FEE.md` と現行コードから、保守売上管理の要件は次の通りです。

- 顧客ごとの月額保守料金を一覧で確認できること
- 0円や休止顧客を除外し、アクティブな保守売上を見られること
- 月次スナップショットを保持し、過去月が後から書き換わらないこと
- 当月だけは API から再同期できること
- 必要に応じて手動修正できること
- サマリーカードと月次推移で経営数値の確認に使えること
- この数値がダッシュボードや見積の資金系指標にも流用されること

## 2. 現仕様

### データ取得と保存

- 月選択時に `maintenance_fee_snapshots` を参照し、対象月が無ければ API から `/customers` を取得して月次スナップショットを生成する
- 当月の demo snapshot は API 再取得で置換される
- 手動再同期は「当月のみ」
- 一覧行は `maintenance_fee_snapshot_items` に保存される

### 画面

- サマリーカード: 月額保守合計 / アクティブ顧客数 / 平均保守金額
- グラフ: 選択月を基準にした過去6ヶ月
- フィルタ: 年月選択、顧客名検索
- 編集: 行の追加 / 更新 / 削除

### 周辺連携

- 見積側の保守売上合計取得も snapshot を最優先で読む
- ダッシュボード集計でも maintenance snapshot を加算している

## 3. 要件と現仕様の差分

| 観点 | 要件/期待 | 現仕様 | 評価 |
| --- | --- | --- | --- |
| 数値の信頼性表示 | 何月の、どのソースの数値か分かる | 画面上に source / 最終同期時刻 / 手修正有無が出ない | 不足 |
| 検索/絞り込み | 顧客名とサポート種別で絞り込み | 顧客名検索のみ UI に存在。`support_type` は backend にあるが UI 未実装 | 不足 |
| 一覧の整列 | 金額順や顧客名順で見たい | 明示ソートなし | 不足 |
| ページング | 件数増でも使える | 全件描画、ページングなし | 不足 |
| エラーハンドリング | API失敗が利用者に分かる | 画面は空表示寄り、ログ中心 | 不足 |
| 修正履歴 | 誰が何を変えたか追える | 変更理由・変更者・変更前後を保持していない | 不足 |
| 再同期運用 | API値との差分確認ができる | 再同期は即上書きで差分確認なし | 不足 |
| 周辺整合 | ダッシュボード/見積と同じ定義 | 取得ロジックが複数箇所に分散 | リスクあり |

## 4. 現状の主要リスク

### 4-1. 画面の合計と表の見え方がずれる

顧客名検索はクライアント側だけで効いている一方、サマリーの `total_fee` は snapshot 全体の合計を表示しています。  
そのため、検索後に「表は数件しか見えていないのに合計が全体のまま」というズレが起きます。

### 4-2. 手修正が監査できない

今は手修正しても、変更理由・変更前・変更者が残りません。  
保守費用は契約変更や休止/再開が混ざるので、証跡が無いと月次確認で揉めます。

### 4-3. 同じ意味のロジックが複数箇所にある

保守売上の取り込み・判定が `MaintenanceFeeController`、`EstimateController`、ダッシュボード集計側へ分散しています。  
1 箇所の仕様変更が他に反映されず、今回のような dev / local / prod 差分を再発させやすい構成です。

### 4-4. snapshot item に業務キーが無い

現行 item は `customer_name` 中心で、外部顧客IDを保持していません。  
名前変更や同名顧客があると、再同期差分や履歴追跡が弱いです。

## 5. ブラッシュアップ提案

## P1. まず入れるべき改善

### P1-1. 数値の状態表示を追加する

画面上部に次を表示する。

- 対象月
- データソース `API / 手修正 / 混在`
- 最終同期日時
- 最終更新者
- 手修正件数

これで「この数字を信用していいか」が一目で分かります。

### P1-2. フィルタとサマリーの整合を取る

- `support_type` フィルタ UI を追加
- ソートを追加
  - 既定: 月額保守料金の降順
  - 任意: 顧客名昇順 / 更新日時降順
- サマリーに `全体` と `表示中` を分ける

これで、一覧を絞ったときの見え方と合計の意味が揃います。

### P1-3. APIエラーと空状態を分ける

- API失敗時は警告バナーと再試行導線を表示
- 「0件」と「取得失敗」を別メッセージにする

今は利用者から見ると両方とも「何も無い」に見えやすいです。

### P1-4. 取り込みロジックを Service 化する

`MaintenanceFeeSyncService` のような 1 箇所へ集約し、次を共通化する。

- API取得
- 休止/0円除外
- snapshot 作成
- 再同期時の上書き
- source 判定

利用先:

- `MaintenanceFeeController`
- `EstimateController`
- バックフィル/再同期コマンド
- ダッシュボード集計の前処理

## P2. 運用を安定させる改善

### P2-1. 修正履歴テーブルを追加する

最低限、次を残す。

- snapshot_item_id
- customer_external_id
- 変更前金額
- 変更後金額
- 変更理由
- 操作者
- 実施日時
- 変更種別 `api_sync / manual_edit / delete / add`

これで「なぜ API 値と違うのか」を後から説明できます。

### P2-2. 再同期の差分プレビューを追加する

「当月を再同期」で即上書きせず、先に差分一覧を見せる。

- 新規顧客
- 消えた顧客
- 金額変更
- ステータス変更

承認後に反映する形にすると、誤上書きを減らせます。

### P2-3. 月次確定フローを入れる

- 当月: 再同期可能
- 締め済み月: 原則 read only
- 例外修正: 権限付きで reopen

過去月も自由に編集できる状態は、経営数値としては危険です。

## P3. 使いやすさを上げる改善

### P3-1. 変化の見える化

- 前月比増減
- 顧客増減件数
- 単価変更件数
- 解約/休止候補

単なる一覧から「気づける画面」に変えられます。

### P3-2. CSV出力

月次レビューや経理確認用に、対象月の明細と差分履歴を CSV で落とせるようにする。

### P3-3. 編集権限の分離

- 閲覧のみ
- 当月修正可
- 過去月修正可
- 月次確定可

現状は編集操作のガードが薄いです。

## 6. 推奨実装順

### フェーズ1

- source / 最終同期 / 手修正件数の表示
- `support_type` フィルタ UI
- 一覧ソート
- サマリーの `全体 / 表示中` 分離
- APIエラーバナー
- sync logic の Service 化

### フェーズ2

- 修正履歴テーブル
- 再同期差分プレビュー
- snapshot item への外部顧客ID保持

### フェーズ3

- 月次確定フロー
- 権限分離
- CSV出力
- 前月比や解約候補の可視化

## 7. 実装前に決めるべきこと

- 保守売上の `gross = sales` を今後も固定で扱うか
- 手修正は「API値の例外補正」なのか「正式な月次確定値」なのか
- 過去月修正を誰まで許可するか
- 顧客の業務キーは API のどのフィールドを正とするか
- ダッシュボードと見積で使う値を「常に snapshot 正本」に統一するか

## 8. 受け入れ確認の観点

- 同じ月を local / dev / prod で見たとき、source と最終同期時刻が説明できる
- 当月再同期で、変更件数と変更理由が追える
- 顧客名検索やサポート種別絞り込み時に、表とサマリーの意味が一致する
- ダッシュボード / 見積 / 保守売上管理で月額保守合計の定義が一致する
- 過去月は意図せず書き換わらない
